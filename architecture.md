# Архитектура масштабированного решения

## Задача
Масштабирование парсинга позиций товаров на Ozon до объёма **500 товаров × 20 поисковых запросов = 10 000 проверок в день**.  
Требуется:
- Хранение результатов в PostgreSQL.
- Отправка алерта в Telegram при падении позиции товара ниже 50.
- Устойчивость к блокировкам со стороны Ozon.

## Используемые технологии

| Компонент       | Выбор                    | Обоснование                                                                                          |
|-----------------|--------------------------|------------------------------------------------------------------------------------------------------|
| Язык            | Python 3.10+             | Основной язык разработки, богатая экосистема для парсинга и веб-автоматизации.                       |
| Парсинг         | Playwright (асинхронный) | Поддержка headless-браузера, эмуляция реального пользователя, устойчивость к динамическому контенту. |
| Очередь задач   | Celery + Redis           | Распределение нагрузки, планирование запусков, повторные попытки.                                    |
| Планировщик     | Celery Beat              | Гибкие интервалы запуска (например, каждые 15 минут для разных товаров).                             |
| Хранение        | PostgreSQL               | Надёжное реляционное хранилище для истории позиций, простота запросов.                               |
| Алерты          | Telegram Bot API         | Быстрая доставка уведомлений, простота интеграции.                                                   |
| Прокси          | BrightData / ProxyCrawl  | Резидентные и мобильные IP с ротацией, высокая анонимность, низкий процент блокировок.               |
| Контейнеризация | Docker                   | Единообразие окружения, лёгкое масштабирование воркеров.                                             |
| Мониторинг      | Prometheus + Grafana     | Отслеживание ошибок, задержек, количества проверок.                                                  |

## Схема

1. **Celery Beat** генерирует задания по расписанию (например, каждые 30 минут для каждого товара).
2. **Redis** выступает брокером сообщений.
3. **Celery Workers** - воркеры, каждый:
   - Выбирает свободный прокси из пула.
   - Запускает Playwright с этим прокси, выполняя парсинг.
   - Сохраняет результат (позицию, страницу, время) в PostgreSQL.
   - Проверяет условие: если позиция ниже 50, отправляет алерт в Telegram.
4. **PostgreSQL** хранит таблицы:
   - `products` (id, sku, query, created_at)
   - `checks` (id, product_id, position, page, total_checked, timestamp, is_alert_sent)
5. **Telegram Bot** отправляет уведомления.

## Обеспечение устойчивости

### Прокси-пул
- Используем **50 резидентных IP** (например, от BrightData) с ротацией после каждого запроса или каждых 5 запросов.
- При ошибке (капча, таймаут) - автоматическая смена прокси и повтор задачи.
- Бюджет: $30-50/мес.

### Rate limiting
- На уровне кода: не более 1 запроса в 5-10 секунд на один прокси.
- Случайные задержки между запросами (3-6 сек на страницу, 5-10 сек между страницами).
- Распределение запусков во времени: не все товары проверяются одновременно, а с разными интервалами (с помощью Celery Beat).

### Обход блокировок
- Используем расширенные заголовки (Accept-Language, Referer, DNT, Sec-Ch-UA и др.).
- Ротация User-Agent (10-20 вариантов).
- Сохранение cookies между запусками для имитации постоянного пользователя.
- Обнаружение капчи: при её появлении задача откладывается с экспоненциальной задержкой и меняется прокси.

### Обработка ошибок
- Повторные попытки (3 попытки) с экспоненциальной задержкой.
- Логирование всех ошибок в отдельную таблицу для анализа.
- Мониторинг: при превышении порога ошибок (например, 5% от всех задач) - оповещение в Telegram.

## Риски и их снижение

| Риск                                       | Снижение                                                                                                              |
|--------------------------------------------|-----------------------------------------------------------------------------------------------------------------------|
| **Блокировка всех прокси**                 | Использовать несколько поставщиков прокси (например, BrightData + ProxyCrawl), иметь резервный пул бесплатных прокси. |
| **Изменение структуры страницы Ozon**      | Автоматическое тестирование селекторов в CI/CD; при падении тестов - уведомление разработчиков.                       |
| **Высокая нагрузка на БД**                 | Партиционирование таблицы `checks` по времени, индексы по `(product_id, timestamp)`.                                  |
| **Падение воркеров**                       | Docker + оркестрация (Kubernetes или Docker Swarm) - автоматический перезапуск. Supervisor для простого случая.       |
| **Капча, требующая ручного вмешательства** | Использовать сервис распознавания капчи (Anti-Captcha, 2Captcha) для автоматического прохождения.                     |

## Бюджет на инфраструктуру (месяц)

| Статья расходов                                                   | Пример                  | Стоимость    |
|-------------------------------------------------------------------|-------------------------|--------------|
| **Выделенный сервер** (4 vCPU, 8 GB RAM, 100 GB SSD)              | DigitalOcean / Hetzner  | $40-60       |
| **Управляемая PostgreSQL** (или свой экземпляр на том же сервере) | -                       | $0-20        |
| **Управляемый Redis** (или свой)                                  | -                       | $0-15        |
| **Прокси-пул** (50 резидентных IP, ротация)                       | BrightData / ProxyCrawl | $30-50       |
| **Мониторинг** (Prometheus + Grafana)                             | -                       | $0-10        |
| **Итого**                                                         | -                       | **$100-150** |

При развёртывании на одном сервере (PostgreSQL и Redis на той же машине) можно уложиться в $70-90, но это повышает риски. В production рекомендуется использовать managed-сервисы для отказоустойчивости.

## Заключение

Предложенная архитектура позволяет масштабировать парсинг до 10 000 проверок в день, обеспечивая устойчивость через прокси-пул, распределение нагрузки, повторные попытки и мониторинг. Использование Celery и Playwright даёт гибкость и надёжность. Бюджет $100-150/мес - реалистичен для подобного объёма.

--- 
*Данная архитектура может быть реализована как набор Docker-контейнеров, развёрнутых на одном сервере для старта, с возможностью горизонтального масштабирования воркеров при росте нагрузки.*
